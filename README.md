# ScalpelMod

Professional tools. Questionable applications.

ScalpelMod is a systems-focused Minecraft mod built around professional utility tools being used beyond their intended context.

This is not a weapon mod.  
This is a toolbox under pressure.

---

## Core Philosophy

- Tools first, weapons second  
- Serious tone  
- Tools used beyond their intended context  
- Everything has a purpose  
- Everything connects  
- Quality over quantity  

If it does not connect, it does not belong.

---

## Mod Scope

ScalpelMod includes:

- Tools  
- Blocks  
- Mobs  
- Structures  
- Interconnected systems  

All content should feel like it came from:

- a lab  
- a job site  
- a facility  
- a professional environment  

---

## Design Rules

### Interconnection

Every feature must:

- connect to something  
- be used by something  
- contribute to a system  

No standalone content is allowed.

---

### Utility First

Every addition must answer:

"What is this used for outside of combat?"

---

### Tone

All content must follow:

Dead serious presentation.  
Tools used beyond their intended context.

---

## Default Asset Format

### Core Principle

Consistency of pixel size matters more than resolution.

---

### Base Standard

- Default assets are 16x16  
- Pixel density must remain consistent  

---

### Allowed Scaling

Higher resolution textures (e.g. 32x32, 64x64) are allowed only if:

- used for larger objects  
- pixel size visually matches 16x16 density  
- no additional detail density is introduced  

---

### Not Allowed

- higher resolution used to add detail  
- mixed pixel densities  
- HD textures in base mod  

---

### Simple Rule

Higher resolution is for size, not detail.

---

## Inventory Display

### Core Principle

Follow vanilla behavior unless there is a clear reason not to.

---

### Items

All items must use 2D sprite-based inventory icons.

Applies to:
- tools  
- equipment  
- utility items  
- 3D modeled items  

---

### Blocks

Blocks follow vanilla behavior:

- default: 3D inventory render  
- 2D overrides allowed if appropriate  

---

### Plants

Plant-like blocks must:

- use 2D inventory sprites  
- follow vanilla conventions  

---

### Simple Rule

Items are 2D.  
Blocks follow vanilla.  
Plants stay flat.

---

## Asset Rules

### Internal Use

Once an asset is included in a published version:

- it becomes a community asset  
- it may be reused within the project  
- original creator must be credited  
- it cannot be revoked  

---

### External Use

Assets may not be used outside this project without permission from the original creator.

---

### Duplication

Contributors may:

- duplicate any asset  
- modify duplicates  
- create variants  

This is allowed regardless of license.

---

### Modification

Original assets may not be edited directly without permission.

Instead:

- duplicate  
- modify the duplicate  
- create a variant  

---

### Replacement

New content is always allowed.

Replacing existing content requires approval.

Includes:

- textures  
- models  
- linked systems  

---

### Linked Assets

Creating variants is always allowed.

Redirecting existing usage is considered replacement and requires approval.

---

## Resource Packs

### General Rule

Resource packs are always allowed.

They may:

- fully retexture the mod  
- change style  
- reinterpret visuals  

No permission required.

---

### Base Mod Style

The core mod must remain:

- consistent  
- readable  
- professional  

Out-of-style visuals must not exist in the base mod.

---

### Promotion to Default

A resource pack may become default only if:

- invited by maintainers  
- creator agrees  
- textures are integrated into the mod  
- old textures moved to a legacy pack  

---

## Visual Style

### Base Requirements

- clean  
- functional  
- professional  

---

### Not Allowed

- exaggerated visuals that break style  
- hyper-realistic visuals  
- meme textures  

---

### Allowed via Resource Packs

- alternate art styles

## Safety Rules

Strictly prohibited:

- malware  
- obfuscated code  
- external downloads  
- hidden behavior  
- unauthorized system access  

All changes must be:

- readable  
- reviewable  
- testable  

---

## Quality Standard

Quality over quantity.

All additions must:

- match style  
- have purpose  
- connect to systems  
- be polished  

---

## Content Restrictions

### Not Allowed

- gore  
- graphic or realistic violence  
- disturbing content  
- inappropriate material  

---

### Allowed

- stylized, non-graphic interactions  
- gameplay-focused effects  

---

### Visual Tone Rule

If it looks realistic, it is not allowed.  
If it is clearly stylized, it is acceptable.

---

## Lore

Lore is optional and must remain subtle.

- no required story  
- no large exposition  
- no gameplay dependence  

---

### Consistency

- one continuous plot line  
- no contradictions  
- no retcons  
- no alternate timelines  

---

## Enforcement

### Asset Duplication Abuse

Trivial edits do not create a valid new asset.

---

### Replacement Bypass

Indirect replacement via redirects is not allowed.

---

### Resource Pack Misuse

Not allowed:

- redistributing core assets externally  
- bypassing licensing  
- impersonating official versions  

---

### License Conflicts

Unclear or conflicting licenses may be rejected.

---

### Credit Integrity

- creators must be credited  
- attribution cannot be removed  
- derivative work must acknowledge sources  

---

### Asset Identity

Assets must remain traceable.

Not allowed:

- removing IDs  
- hiding authorship  
- renaming to obscure origin  

---

### Code Safety

Not allowed:

- dynamic code execution  
- runtime downloads  
- hidden logic  
- misleading structure  

---

### Maintainer Authority

Maintainers have final say on:

- approvals  
- rejections  
- reversions  

Community input informs decisions but does not override maintainers.

---

## Final Principle

Add anything.  
Replace carefully.  
Everything must connect.  
Everything must have a purpose.  
Style remains consistent.
